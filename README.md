# SchoolBridge OCR Lab

> 학교 안내문 사진 입력의 가능성을 수치로 검증한 독립 실험실

![SchoolBridge OCR pipeline](pipeline.png)

---

## 왜 이게 필요했나

SchoolBridge는 단순 번역기가 아닙니다.

**학교 안내문을 읽기 어려운 사람이 핵심 정보를 이해하고 행동할 수 있게 돕는 서비스**입니다.

그런데 서비스가 가장 필요한 사람일수록 파일이 없습니다.

```
파일이 없는 사용자
→ 종이 안내문만 있는 다문화 학부모
→ 카카오톡으로 받은 이미지 스크린샷
→ 학교 앞 게시판 사진
```

이런 사용자에게 "PDF로 올려주세요"라고 하면 서비스가 끊깁니다.

> 디지털 파일을 다루기 어려운 분들이 타겟인데,
> PDF만 받으면 정작 그분들이 못 씁니다.

그래서 **사진으로도 입력받을 수 있는지** 수치로 검증했습니다.

## 강사님 피드백 반영 (2026-04-30)

### 칭찬받은 부분

- 파이프라인 분업 구조 (윤정 → 경이 → 세종) 자체는 긍정 평가
- B2G(학교·교육청 납품) 비즈니스 모델 타당성 인정

### 지적받은 부분

- "텍스트 추출을 완벽하게 뽑으려다 시간 다 간다"
- "특수기호(▪ □ ■) 정제에 집착하면 핵심 기능을 못 만든다"
- "최종적으로 얼마나 깔끔하게 번역/음성으로 나오는가가 중요"

### 이 실험이 피드백에 부합하는 이유

**1. 특수기호 전처리는 이미 파이프라인에 들어가 있습니다**

Quality Gate(`ocr_quality_gate.py`)와 Model Input Builder(`model_input_builder.py`)가
이미 다음을 처리합니다:

- 의미 없는 기호 줄 자동 제거 (`_is_noise_line`: 의미 문자 비율 < 25% 줄 삭제)
- 가비지 문자 비율 측정 및 경고 (`garbage_ratio > 0.10` 시 auto_pass 거부)
- 중복 줄 제거, OCR 자간 오류 정규화

즉, 강사님이 지적한 "짝대기 빼는 것"이 이미 코드로 구현되어 있습니다.

**2. CER이 높아도 F1이 높으면 파이프라인이 흡수합니다**

```
ML Kit CER 35%  →  표 셀 순서 혼재가 주원인
ML Kit F1  0.95~0.97  →  글자 자체 인식은 양호
```

우리 파이프라인은 OCR 원문을 바로 번역하지 않습니다.

```
OCR 결과
  → 윤정님 할일 추출 (필요한 문장만 뽑음 — 표 순서 노이즈 대부분 탈락)
  → 경이님 분류 (한 번 더 필터링)
  → 세종 번역 (정제된 액션 문장만 입력)
```

단순 OCR 품질이 아니라 **파이프라인 전체 품질**로 판단해야 합니다.

## 회의 피드백 반영 — 방향 전환 (2026-05-06)

2026-05-06 회의에서 받은 핵심 피드백은 다음과 같습니다.

> "텍스트만 뽑아서 요약하는 방식은 ChatGPT에 PDF를 올리는 것과 차별성이 약하다.
> 원본 문서의 위치 정보를 보존하고, 중요한 정보를 원본 위에 하이라이트하는 방향이 더 적절하다."

따라서 OCR Lab의 역할도 단순한 `이미지 → 텍스트` 검증에서 다음 방향으로 확장합니다.

```
원본 문서 / 사진
  → OCR or PDF layout parser
  → text + bounding box + page 정보 추출
  → 윤정님 할일 추출 / 경이님 카테고리 분류
  → 중요 문장과 원본 좌표 매핑
  → 원본 문서 위에 일정·비용·준비물·제출 항목 하이라이트
```

즉 SchoolBridge의 포지션은 단순 요약기가 아니라,
**원본 가정통신문 위에 행동 정보를 표시하는 시각적 보조 서비스**입니다.

### OCR 연구가 계속 필요한 이유

- ML Kit `TextBlock` / `Line` 결과는 텍스트뿐 아니라 bounding box를 제공할 수 있음
- 기존 표 감지/2-pass OCR 코드는 표 영역 하이라이트로 확장 가능
- Quality Gate는 OCR 결과를 무조건 믿지 않고 재촬영/확인 UX로 연결하는 안전장치가 됨
- 사진 입력은 디지털 파일이 없는 학부모를 위한 필수 입력 경로

### 다음 개발 방향

| 우선순위 | 작업 | 목적 |
|---|---|---|
| 1 | ML Kit 결과에서 block/line bbox 저장 | 원본 이미지 하이라이트 기반 확보 |
| 2 | PDF는 OCR보다 pdfplumber 좌표 기반 layout parser 우선 | 텍스트 PDF의 위치 정보 보존 |
| 3 | HWP/HWPX는 ODT/XML 구조 기반 table row/cell 복원 | HWP 표 구조 안정화 |
| 4 | 추출 모델 결과와 bbox 재매핑 | 할 일/비용/일정 영역 시각화 |
| 5 | 앱에서 원본 문서 위 하이라이트 표시 | ChatGPT 단순 요약과 차별화 |

## 팀 파이프라인에서의 위치

```
[기본 경로 — 변경 없음]

HWP / PDF / TXT
  → 태수님 parser  →  clean_text
  → 윤정님 할일 추출  →  경이님 분류
  → 세종 NLLB 번역 / TTS
```

```
[OCR Lab이 검증한 보조 경로]

사진 / 스캔 이미지
  → Android ML Kit OCR
  → Quality Gate  ←── 이게 핵심
       ├─ 통과 (overall ≥ 0.90)  →  clean_text 후보  →  기본 경로 연결 가능
       └─ 미달                   →  "원문 확인 필요" 안내
```

OCR Lab은 **기본 경로를 대체하지 않습니다.**
태수님 파트를 건드리지 않았고, 팀 메인 레포에도 코드를 넣지 않았습니다.
별도 실험실에서 가능성과 한계를 수치로 확인했습니다.

## 실험 결과 요약

### 모델 비교 (카메라 입력 기준)

| 모델 | 평균 CER | F1 | 비고 |
|---|---|---|---|
| **ML Kit Korean** | **35.3%** | **0.95~0.97** | 정상 동작 |
| EasyOCR | 98.2% | — | 한국어 인식 실패 |
| Tesseract | 96.8% | — | 한국어 인식 실패 |
| PaddleOCR | — | — | 환경 호환성 오류로 실행 불가 |

한국어 특화 모델(ML Kit Korean) 외 대안 없음이 실험으로 확인됐습니다.

### ML Kit 조건별 상세

| 조건 | CER | WER | 자모 CER | F1 |
|---|---|---|---|---|
| A: 깨끗한 PDF | 36.1% | 44.0% | 34.4% | 0.9651 |
| B: 카메라 정면 | 36.8% | 41.8% | 34.8% | 0.9747 |
| C: 카메라 기울임 | **31.0%** | **30.4%** | **24.3%** | 0.9655 |
| D: 구겨진 인쇄물 | 37.3% | 42.1% | 31.1% | 0.9566 |

> CER이 높은 주원인은 **표 셀 순서 혼재**입니다.
> F1이 0.95 이상이라는 건 글자 자체 인식은 양호하다는 뜻입니다.
> 윤정님 추출 단계가 표 노이즈를 걸러주므로 파이프라인 전체로는 가능성 있습니다.

### 발표할 때 이렇게 말하면 됩니다

> "Android ML Kit OCR은 F1 0.95 이상으로 핵심 단어 탐지는 가능했습니다.
> CER이 높은 주원인은 표 셀 순서 혼재인데, 이는 윤정님 추출 단계에서 필터링됩니다.
> 디지털 PDF는 pdfplumber, 종이·카메라는 ML Kit으로 입력 경로를 분리하는 것이 최적 설계입니다.
> 서비스 타겟인 다문화 학부모분들은 카메라 입력밖에 없는 경우가 많기 때문에,
> OCR 보조 경로는 타겟 사용자에게 필수입니다."

---

## 이게 실패가 아닌 이유

강사님이 OCR을 말린 이유는 이것이었을 겁니다.

> OCR을 MVP 핵심 기능으로 넣다가 프로젝트 전체를 망치지 말라

이번에 한 작업은 다릅니다.

- 팀 메인 레포를 건드리지 않았습니다
- 독립 실험실에서만 진행했습니다
- 실패/한계까지 수치로 남겼습니다
- Quality Gate로 안전장치를 만들었습니다
- 특수기호 전처리도 이미 구현했습니다

**무리한 확장이 아니라, 리스크를 분리한 검증 작업입니다.**

## 구현된 것들

### 1. Tesseract OCR Lab

로컬에서 돌아가는 OCR 파이프라인입니다.

```
사진 입력
  → 자동 회전 보정 (0 / 90 / 180 / 270도 후보 평가)
  → 전처리 6종 variant (grayscale / threshold / warped 등)
  → 표/박스 영역 crop + 2배 / 3배 확대 OCR
  → 셀 단위 구조 복원 (행/열 분리, 패턴 추출)
  → result.json 저장
```

### 2. Quality Gate

OCR 결과의 신뢰도를 자동으로 평가합니다.

```python
# 4개 점수를 합산해 overall_score 계산
text_quality  # 쓰레기 문자 비율, 의미 있는 단어 밀도
pattern       # URL / 전화번호 / 날짜 / 시간 / 금액 탐지 여부
table         # 표 후보 탐지, 셀 OCR 성공률
coherence     # 문장 구조, 줄 수, 문서 길이
```

결과 분기:
```
overall ≥ 0.90  →  verified_text.txt  (자동 통과 후보)
overall < 0.90  →  review_text.txt    (원문 확인 필요)
```

### 3. Model Input Builder

OCR 결과나 PDF 변환 텍스트를 후속 모델 입력 전 정제합니다.

특수기호 처리 포함:
- 의미 문자 비율 25% 미만 줄 자동 제거
- 중복 줄 제거
- OCR 자간 오류 정규화 (예: `학 년 도` → `학년도`)

```powershell
python tools\build_model_input.py --text "OCR\meal_notice.txt" --output outputs\model_input
```

## 한 줄 결론

> OCR은 완전 자동 입력으로 쓰기엔 아직 부족하지만,
> Quality Gate + 파이프라인 필터링(윤정→경이)을 거치면
> **사진 입력 보조 경로**로 가능성이 있다.
> 서비스 타겟인 다문화 학부모는 카메라 입력밖에 없는 경우가 많기 때문에
> 이 경로는 선택이 아니라 필수다.
